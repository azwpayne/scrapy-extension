## Root cause (cross-cutting)

The H commit (`08f2a69`) shipped the real at-least-once fix for Kafka/RabbitMQ and, as scope-creep
cleanup, **removed the E1 fail-fast guard** that previously protected *all* ack-using backends.
The removal was predicated on "the in-flight-set makes ack correct" — true only for the two
backends that got the fix. SQS/Pulsar kept single-slot ack and lost their only safety net, and the
scheduler docstring now over-claims universal concurrent-ack correctness. This is a **scope-creep
regression**, not a new bug. The same pattern (a fix for one cohort silently widening blast radius
for another) recurs at the pipeline-error seam (catch-all helps transient errors but silently
absorbs persistent outages) and the dedup lifecycle (global default key helps first-run simplicity
but silently breaks resume).
